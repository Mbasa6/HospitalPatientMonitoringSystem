## 1. Overview

The HPMS domain model captures the core entities, their attributes, responsibilities, and relationships that form the backbone of the hospital patient monitoring system. These entities were identified from functional requirements (Assignment 4), use cases (Assignment 5), and state diagrams (Assignment 8).

---

## 2. Domain Entities

| Entity | Attributes | Methods | Relationships |
|---|---|---|---|
| **Patient** | patientId, firstName, lastName, dateOfBirth, diagnosis, status, admittedAt, dischargedAt | register(), discharge(), assignToWard(), getVitalHistory() | Assigned to one Ward; has many VitalReadings; has many Alerts; has many AlertThresholds |
| **VitalReading** | readingId, patientId, heartRate, bloodPressureSystolic, bloodPressureDiastolic, oxygenSaturation, temperature, respiratoryRate, recordedAt, sourceDeviceId | save(), validate(), getLatest() | Belongs to one Patient; may trigger one Alert |
| **Alert** | alertId, patientId, vitalReadingId, vitalType, triggeredValue, severity, status, triggeredAt, acknowledgedAt, acknowledgedById | trigger(), acknowledge(), escalate(), close() | Belongs to one Patient; linked to one VitalReading; acknowledged by one User |
| **AlertThreshold** | thresholdId, patientId, vitalType, minValue, maxValue, severity, isPatientOverride, createdById | validate(), applyToPatient(), revertToDefault() | Belongs to one Patient (override) or system-wide; created by one User |
| **User** | userId, firstName, lastName, email, passwordHash, role, wardId, isActive, lastLoginAt | login(), logout(), resetPassword(), deactivate() | Has one Role; assigned to one Ward; may acknowledge many Alerts |
| **Ward** | wardId, name, floor, capacity, currentOccupancy, status | addPatient(), removePatient(), getActiveAlerts(), getSummary() | Contains many Patients; has many Users assigned; generates Reports |
| **Report** | reportId, patientId, generatedById, dateRangeStart, dateRangeEnd, generatedAt, filePath, status | generate(), download(), archive() | Belongs to one Patient; generated by one User |
| **AuditLog** | logId, userId, action, targetEntity, targetId, timestamp, ipAddress | create(), export() | Created by any User action; references any entity |

---

## 3. Relationships

| Relationship | Type | Description |
|---|---|---|
| Patient — VitalReading | Composition | A patient owns their vital readings; readings cannot exist without a patient |
| Patient — Alert | Composition | Alerts are generated from a patient's vitals; they belong to that patient |
| Patient — AlertThreshold | Aggregation | Thresholds can exist as system defaults independent of a specific patient |
| VitalReading — Alert | Association | A vital reading may trigger one alert; the alert references the reading |
| User — Alert | Association | A user (nurse) acknowledges alerts; alerts reference the acknowledging user |
| User — Ward | Association | Users are assigned to a ward; a ward has many users |
| Ward — Patient | Aggregation | A ward contains patients; patients can be reassigned to different wards |
| User — Report | Association | A user generates a report; report references the generating user |
| Patient — Report | Composition | Reports belong to a patient and cannot exist independently |
| User — AuditLog | Association | Every user action generates audit log entries |

## 4. Business Rules

| Rule | Description | Maps To |
|---|---|---|
| BR-01 | A patient must be assigned to exactly one ward and one bed at any time | FR-01 |
| BR-02 | A vital reading must include all five vital signs to be valid | FR-02 |
| BR-03 | An alert is automatically generated within 1 second of a threshold breach | FR-05 |
| BR-04 | An unacknowledged critical alert must be escalated after 5 minutes | FR-06 |
| BR-05 | Only a Doctor or Admin can modify alert thresholds | FR-04 |
| BR-06 | A user account is locked after 5 failed login attempts within 1 minute | NFR-SE03 |
| BR-07 | Audit log entries cannot be modified or deleted by any user role | FR-12 |
| BR-08 | A report can only be generated for a patient by an authorized Doctor or Admin | FR-10 |
| BR-09 | A ward cannot admit patients beyond its maximum bed capacity | FR-11 |
| BR-10 | Alert thresholds revert to system defaults when a patient is discharged | FR-04 |
