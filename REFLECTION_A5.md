# Reflection

Translating system requirements into use cases and test cases presented several challenges. One major challenge was ensuring that all functional requirements were properly represented as user interactions. Functional requirements describe system behavior, while use cases focus on how users engage with the system, requiring careful interpretation.

Another challenge was determining the correct level of detail. Some use cases were initially too broad, making it difficult to define clear steps. Refining these into smaller, structured flows helped improve clarity and testability.

Aligning stakeholders with use cases was also complex. Each stakeholder has unique needs, and ensuring these were reflected accurately required revisiting the stakeholder analysis from the previous assignment.

Creating test cases introduced further challenges, especially for non-functional requirements like performance and security. These required defining measurable criteria, which is more abstract than functional testing.

Maintaining consistency across all assignments was critical. The system defined earlier needed to align with the use cases and test cases to ensure coherence.

Overall, this process improved understanding of how requirements are transformed into system interactions and validated through testing. It reinforced the importance of structured documentation and traceability in software engineering.
